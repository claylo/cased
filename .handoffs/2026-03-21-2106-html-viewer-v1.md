# Handoff: HTML viewer v1 — working but buggy

**Date:** 2026-03-21
**Branch:** `feat/html-viewer` (16 commits ahead of main)
**State:** Yellow

> Yellow = core pipeline works end-to-end, but multiple visual/interaction bugs need fixing before this is presentable.

## Where things stand

The HTML viewer build pipeline is complete: `node src/viewer/build-report.mjs <audit-dir>` reads YAML, renders semantic HTML, inlines CSS/fonts/JS, and writes a single self-contained `report.html` (~246KB). rough.js terrain map draws, rough-notation annotations fire on scroll, and presentation mode toggles. But the output has visual bugs that need a browser-in-hand polish pass.

**The superpowers-chrome browser has the report loaded.** Start by navigating to the report and using `use_browser` to inspect — do not rebuild blindly.

## Decisions made

- **YAML is canonical** — HTML viewer, README.md, and PDF are all rendering frontends of `recon.yaml` + `findings.yaml`. Spec at `record/superpowers/specs/2026-03-21-html-viewer-design.md`.
- **JSON schemas are prescriptive** — `src/cased/references/{findings,recon}.schema.json` define the contract. If example data doesn't conform, fix the data.
- **Skill ships pre-built JS** — rolldown bundles rough.js + rough-notation + viewer logic at dev time. The shipped skill only needs `node` to run `build-report.js`.
- **Build-time HTML + client-side enhancements** — Content is in the DOM (searchable, printable). JS adds rough.js terrain map, annotations, sparklines, and presentation mode.

## What's next

   - Summary pills ("1 significant", "2 moderate") look clickable but aren't — either make them filter/scroll to findings or remove the interactive affordance
   - Terrain map labels show `mod.rs` twice (ambiguous) — use crate-relative paths like `curate/mod.rs`
   - Terrain map dependency lines are nearly invisible (`strokeWidth: 0.8`, light gray)
   - Terrain map needs a legend explaining what red stroke means
   - Presentation mode slides need visual verification — were fixed in `a56ee46` but not browser-tested

2. **Rendering polish:**
   - `renderProse()` handles inline backtick→`<code>` but doesn't handle other markdown (bold, links in remediation text)
   - Evidence code blocks could use basic syntax highlighting (or at minimum a language label from the file extension)
   - The `current-repo-review` title should probably be titlecased in the H1

3. **Test the bundled skill path:** `node dist/build-report.js` works but `src/cased/scripts/build-report.js` (the skill-shipped version) hasn't been tested end-to-end.

## Landmines

- **`viewer.js` name collision** — `src/viewer/viewer.js` (ESM source) and `dist/viewer.js` (IIFE bundle) share a name. The CLI uses `existsSync` fallback logic (`src/viewer/build-report.mjs:284-290`) to pick the right one. If you add a `viewer.js` file anywhere unexpected, the wrong one gets inlined.
- **rolldown CJS bundling** — `build-report.mjs` must not use top-level `await` (CJS doesn't support it). A subagent introduced one; it was removed in `75a1656`.
- **`renderNarrative` signature changed** — now takes `(narrative, slugToTitle)` not just `(narrative)`. The test at `test/build-report.test.mjs` calls it with one arg (will still work because JS, but the test should be updated).
- **`scripts/build-viewer.sh` copies assets to both `dist/` and `src/cased/`** — if you change the asset list, update both copy blocks.
- **No `monthly_commits` in example data** — sparklines are implemented but untestable with the current example YAML. Add synthetic data to test.
