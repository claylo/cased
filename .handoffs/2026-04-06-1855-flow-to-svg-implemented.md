## Where things stand

`src/viewer/flow-to-svg.js` is complete for happy-path (spine-only) flow diagrams. It's a pure function (zero dependencies) imported by `build-report.mjs` at build time, producing inline SVG in narrative sections. The example audit renders a 6-step vertical flow diagram with two annotated findings, a chain reference, and short label overrides. Vertical (>4 steps) and horizontal (<=4 steps) layouts both work.

## Decisions made

- **Direct SVG generation, not Pikchr** — Pikchr was ruled out due to mixed-size text alignment bugs (`small` changes bounding box computation, breaking label positioning). See `record/superpowers/specs/2026-04-06-flow-to-svg-design.md`.
- **Colored connector stems, not annotation arrows** — stroke weight and color encode concern severity (data-ink, not decoration). Visual language defined in `.superpowers/brainstorm/65203-1774280995/symbol-style-v2.html`.
- **Label overrides via string-or-object findings entries** — `findings: ['slug']` uses `finding.title`; `findings: [{slug, label}]` uses the short label. Backward compatible.
- **Per-narrative diagrams** — each narrative's `flow` array produces its own diagram, placed after the thesis paragraph. Narratives without `flow` render as before.
- **Horizontal flows span primary + sidebar** — horizontal layout is for short flows (<=4 steps) and is intentionally wider than the text column. Never sidebar-only.

## Off-spine branch implementation notes

Three tightly coupled features, one implementation task:

**`spine: false` rendering:** Currently filtered out at `flow-to-svg.js:89`. Off-spine steps need to appear beside the spine — positioned to the right of the finding annotation area in vertical layout, connected to their parent decision step. Key question: placement geometry (same y as parent decision? offset right by how much?).

**`no:` decision branch connectors:** A `no: reject` field on a decision step means "if no, go to step with id 'reject'." The "yes" path is implicit (next spine step). Renders as a connector from the decision diamond to the off-spine target step, with a "no" label.

**`next:` loop-back arrows:** A `next: token` field overrides default sequential flow, creating a visual arrow going backward up the spine. Typically Manhattan-routed: right to clear the spine, up to target y, left to reconnect. Hardest part is routing around existing elements.

Reference: the auth flow example in `record/superpowers/specs/2026-03-23-flow-diagram-design.md:53-90` defines the schema for all three features.
