# Cased HTML Viewer — Design Spec

## Overview

The cased skill produces structured code audit reports. The primary artifacts are YAML files (`recon.yaml`, `findings.yaml`). The HTML viewer is one of several rendering frontends — alongside `README.md` (GitHub-native markdown) and `report.pdf` (Typst) — that present the audit data to humans.

The HTML viewer is the richest frontend: a single self-contained `.html` file with hand-drawn visual annotations (rough.js, rough-notation), dual reading modes, and embedded fonts. It requires zero external dependencies to open — just a browser.

## Architecture

### Data flow

```
DEV/CI (this repo):
  rolldown bundles src/viewer/*.js + npm deps
        |
        +--→ viewer.js        (client-side IIFE: rough.js + rough-notation + viewer logic)
        +--→ build-report.js  (node script: yaml parser bundled in)
        |
        v
  skill zip ships: build-report.js, viewer.js, template.html, style.css, fonts/

AUDIT TIME (user's project):
  recon.yaml + findings.yaml
        |
        v
  node build-report.js <audit-dir>
        |
        +-- Parse YAML, validate against JSON schemas
        +-- Generate semantic HTML (narratives, findings, code blocks, ledger)
        +-- Assemble template.html + inlined CSS + base64 fonts + viewer.js + JSON data
        |
        v
  report.html (single file, ~300-500KB)
```

### YAML is canonical

The YAML files are the source of truth. Every rendering frontend reads from the same data:

- `README.md` — generated by the skill's Phase 4 (markdown assembly)
- `report.html` — generated by `build-viewer.mjs`
- `report.pdf` — generated by Typst template (if `typst` is on `$PATH`)

Changes to the YAML schemas (`src/cased/references/recon-schema.yaml.md`, `src/cased/references/findings-schema.yaml.md`) are load-bearing for all renderers.

### JSON schemas

Machine-readable JSON Schema files define the expected shape of `recon.yaml` and `findings.yaml` after parsing. These live alongside the existing prose schema docs in `src/cased/references/`:

```
src/cased/references/
  recon-schema.yaml.md          # Prose guidance (narrative, how-to)
  recon.schema.json             # JSON Schema (structural contract)
  findings-schema.yaml.md       # Prose guidance
  findings.schema.json          # JSON Schema (structural contract)
```

The `yaml` devDependency parses YAML to JS objects; the JSON schemas define what those objects look like. The build script validates parsed YAML against the schemas before rendering — catching missing required fields early rather than producing broken HTML.

The prose `.yaml.md` docs remain the primary reference for skill users (how to choose narratives, what makes a good thesis). The JSON schemas are the structural contract for renderers.

## Source tree

Two contexts: the **development repo** (this repo, has npm deps and rolldown) and the **skill directory** (ships as a zip, zero npm install at runtime).

### Development repo

```
src/viewer/
  template.html          # HTML shell with slot markers
  style.css              # Report styling: Atkinson fonts, code blocks, layout,
                         #   scroll mode, presentation mode
  viewer.js              # Client-side JS: rough.js terrain map, rough-notation
                         #   scroll observer, mode switching, slide navigation,
                         #   sparkline rendering
  build-report.mjs       # Report assembly logic: parse YAML, generate HTML,
                         #   inline assets, emit report.html

scripts/
  build-viewer.sh        # Dev build: runs rolldown to produce pre-built artifacts
                         #   for the skill directory

vendor/
  fonts/
    AtkinsonHyperlegibleNextVF-Variable.woff2    # ~77KB
    AtkinsonHyperlegibleMonoVF-Variable.woff2    # ~39KB
```

Dev dependencies (package.json, build-time only):
- `rolldown` — bundles JS into self-contained artifacts
- `roughjs` — hand-drawn canvas/SVG rendering
- `rough-notation` — animated hand-drawn annotations
- `yaml` — YAML parsing (bundled into build-report.js for skill runtime)

### Skill directory (what ships)

```
src/cased/
  scripts/
    build-report.js      # Self-contained node script (yaml parser bundled in
                         #   by rolldown at dev time). No npm install needed.
  templates/
    viewer.js            # Self-contained IIFE (rough.js + rough-notation +
                         #   viewer logic, bundled by rolldown at dev time)
    template.html        # HTML shell with slot markers
    style.css            # Report styling
    fonts/
      AtkinsonHyperlegibleNextVF-Variable.woff2
      AtkinsonHyperlegibleMonoVF-Variable.woff2
```

At audit time: `node src/cased/scripts/build-report.js <audit-dir>` — reads YAML, assembles single-file HTML. Only dependency is node.

## Output HTML structure

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <title>Cased Report: {project} - {date}</title>
  <style>
    @font-face { /* Atkinson Hyperlegible Next, base64 woff2 */ }
    @font-face { /* Atkinson Hyperlegible Mono, base64 woff2 */ }
    /* style.css inlined */
  </style>
</head>
<body data-mode="scroll">

  <nav id="mode-toggle"><!-- scroll | present toggle --></nav>

  <main id="report">
    <header>
      <h1>{project} Audit</h1>
      <p class="meta">{date} . {commit} . {scope}</p>
      <p class="assessment">{opening paragraph}</p>
      <div class="summary-bar"><!-- concern level counts --></div>
    </header>

    <section id="terrain-map">
      <canvas id="terrain-canvas"></canvas>
    </section>

    <!-- Build-time generated from findings.yaml -->
    <section class="narrative" data-slug="{narrative-slug}">
      <h2>{Narrative Title}</h2>
      <p class="thesis"><em>{thesis}</em></p>

      <article class="finding" data-slug="{finding-slug}"
               data-concern="{concern}">
        <h3>{Finding Title}</h3>
        <div class="finding-meta">
          <span class="concern-badge" data-concern="{concern}">{concern}</span>
          <code>{path}:{lines}</code>
          <canvas class="sparkline"
                  data-commits="{monthly_commits JSON}"></canvas>
        </div>
        <div class="mechanism">{mechanism}</div>
        <pre class="evidence"><code>{evidence}</code></pre>
        <div class="chains"><!-- chain links as anchors --></div>
        <div class="remediation">{remediation + code sketch}</div>
      </article>

      <p class="verdict"><em>{verdict}</em></p>
    </section>

    <section id="remediation-ledger">
      <h2>Remediation Ledger</h2>
      <table><!-- one row per finding, grouped by narrative --></table>
    </section>
  </main>

  <div id="slides" hidden>
    <!-- JS builds presentation slides from DOM + JSON data -->
  </div>

  <script id="cased-data" type="application/json">
    { /* recon + findings merged as JSON */ }
  </script>

  <script>/* viewer-bundle.js inlined */</script>
</body>
</html>
```

## Scroll mode

The default reading mode. Content flows as a long-form document.

### Rough-notation annotations

An `IntersectionObserver` watches findings as they scroll into the viewport and triggers rough-notation annotations:

| Selector | Annotation | Color |
|---|---|---|
| `.concern-badge[data-concern="critical"]` | `box` (red, rough) | `#dc2626` |
| `.concern-badge[data-concern="significant"]` | `underline` | `#dc2626` |
| `.concern-badge[data-concern="moderate"]` | `underline` | `#1a1a1a` |
| `pre.evidence` in critical/significant findings | `bracket` (left) | `#dc2626` |
| `a.chain-ref` | `circle` | `#d1d5db` |
| `p.thesis` | `highlight` | very light gray |

Annotations fire once per element (no re-animation on re-scroll). Default rough-notation roughness settings.

### Rough.js elements

- **Terrain map** — draws on page load (above the fold). Rough.js rectangles for modules, lines for edges, stroke weight for finding density. Sized from `recon.yaml` module data. Monochrome palette.
- **Sparklines** — draw when the finding enters viewport (same observer). Rough.js polyline on a tiny canvas, reading `data-commits` attribute. ~80x16px. Data sourced from `findings.yaml` per-finding `temporal.monthly_commits` when available; falls back to `recon.yaml` hotspot data matched by primary location path. Sparkline canvas is omitted if no commit data exists for a finding.

## Presentation mode

Activated via toggle button (top-right) or `p` key. `Escape` returns to scroll mode, scrolling to the source position of the current slide.

### Slide sequence

For each narrative in the report, the slide deck contains:

| Slide type | Content |
|---|---|
| Title | Project name, date, assessment, summary bar |
| Terrain map | Full-width rough.js terrain map (redrawn larger) |
| Narrative intro | Narrative title + thesis |
| Finding (x N) | Full finding: badge, evidence, mechanism, remediation |
| Narrative verdict | Verdict text |
| *(repeat for each narrative)* | |
| Ledger | Remediation table |

### Navigation

- Arrow keys (left/right) or click to advance/retreat
- Slide counter in bottom-right: `3 / 17`
- Rough-notation annotations play on slide entry (same annotation types as scroll mode, triggered by slide transition)

### Slide construction

The `<div id="slides">` container starts empty. On mode switch, JS clones narrative/finding elements from the DOM into slide frames and uses the embedded JSON blob for terrain map data and slide metadata. No separate slide markup in the source — slides are derived at runtime.

## Terrain map rendering

Rough.js draws the terrain map from `recon.yaml` module data:

- **Modules** as rough rectangles, area proportional to `lines` count
- **Edges** as rough lines between connected modules, stroke width proportional to dependency `weight`
- **Finding density** as increased stroke weight on modules with findings (stroke width, not color)
- **Layout** computed in JS (simple force-directed or grid positioning)
- **Palette**: `#1a1a1a` (near-black), `#6b7280` (mid-gray), `#d1d5db` (light-gray), `#dc2626` (accent, sparingly for critical-heavy modules)

## Build pipeline

Two build contexts, two scripts.

### Dev build (`scripts/build-viewer.sh`)

Runs in this repo during development or CI. Produces the pre-built artifacts that ship with the skill. Requires npm deps (rolldown, roughjs, rough-notation, yaml).

```bash
# Produces two self-contained artifacts:
# 1. dist/viewer.js     — client-side IIFE (rough.js + rough-notation + viewer logic)
# 2. dist/build-report.js — node script with yaml parser bundled in
npx rolldown src/viewer/viewer.js --format iife --file dist/viewer.js
npx rolldown src/viewer/build-report.mjs --format cjs --file dist/build-report.js --platform node
```

The justfile wraps this:
```
build-skill:
    scripts/build-viewer.sh
    # copies dist/ artifacts + template.html + style.css + fonts into src/cased/
```

### Report build (`build-report.js`)

Ships with the skill. Runs at audit time in the user's project. Only requires node.

Single entry point: `node build-report.js <audit-directory>`

Steps:
1. Resolve its own directory to find sibling assets (template.html, style.css, viewer.js, fonts/)
2. Read `recon.yaml` and `findings.yaml` from the audit directory
3. Parse YAML (bundled yaml parser), validate against JSON schemas
4. Generate HTML fragments: header, narratives, findings, ledger
5. Merge recon + findings into a JSON blob for client-side features
6. Read `template.html` from its own directory
7. Inline `style.css` into `<style>`
8. Base64-encode woff2 fonts, inject as `@font-face`
9. Inline `viewer.js` into `<script>`
10. Write assembled `report.html` to the audit directory

## Color palette

Consistent across all visual elements:

| Token | Hex | Usage |
|---|---|---|
| near-black | `#1a1a1a` | Primary text, strong strokes |
| mid-gray | `#6b7280` | Secondary text, metadata, moderate annotations |
| light-gray | `#d1d5db` | Backgrounds, subtle annotations, light strokes |
| accent-red | `#dc2626` | Critical/significant badges, brackets, sparingly |
| accent-green | `#059669` | Clean verdicts, sparingly |

Monochrome-first: the report must be fully readable in grayscale. Red and green are emphasis only.

## Typography

- **Body text**: Atkinson Hyperlegible Next (variable, woff2)
- **Code**: Atkinson Hyperlegible Mono (variable, woff2)
- Both embedded as base64 `@font-face` declarations
- Fallback stack: system-ui, sans-serif (body) / ui-monospace, monospace (code)
- Total font payload: ~116KB (woff2 variable fonts)

## Constraints

- **Single file output** — `report.html` must work when double-clicked from a file manager. No external resources.
- **Skill ships pre-built** — rolldown runs in this repo (dev/CI), not at audit time. The skill zip contains self-contained JS artifacts with all dependencies baked in.
- **Node is the only runtime dependency** — `node build-report.js <dir>` is the entire skill-side build. No npm, no npx, no node_modules required in the target project.
- **Grayscale-safe** — all visualizations must be readable without color. Stroke weight and annotation type carry meaning, not color alone.
